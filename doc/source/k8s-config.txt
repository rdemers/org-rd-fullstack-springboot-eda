## ⚙️ Prérequis côté Spring Boot

Pour que les probes fonctionnent bien, assure-toi d’avoir :

```properties
management.endpoints.web.exposure.include=health,info
management.endpoint.health.probes.enabled=true
```

Et idéalement :

```properties
management.endpoint.health.show-details=always
```

Spring Boot expose alors :

* `/actuator/health/liveness`
* `/actuator/health/readiness`

## 🧩 Exemple complet de `deployment.yaml`

Voici un exemple réaliste et prêt pour EKS :

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: springboot-app
spec:
  replicas: 2
  selector:
    matchLabels:
      app: springboot-app
  template:
    metadata:
      labels:
        app: springboot-app
    spec:
      terminationGracePeriodSeconds: 30  # délai de grâce pour shutdown
      containers:
        - name: springboot-app
          image: ton-repo/springboot-app:latest
          ports:
            - containerPort: 8080

          readinessProbe:
            httpGet:
              path: /actuator/health/readiness
              port: 8080
            initialDelaySeconds: 10
            periodSeconds: 5
            failureThreshold: 3
            timeoutSeconds: 2

          livenessProbe:
            httpGet:
              path: /actuator/health/liveness
              port: 8080
            initialDelaySeconds: 20
            periodSeconds: 10
            failureThreshold: 3
            timeoutSeconds: 2

          lifecycle:
            preStop:
              exec:
                command: ["sh", "-c", "sleep 10"]

          resources:
            requests:
              cpu: "250m"
              memory: "512Mi"
            limits:
              cpu: "500m"
              memory: "1Gi"
```

---

## 🧠 Explication rapide (utile en prod)

### 🔹 `terminationGracePeriodSeconds`

* Kubernetes attend avant de tuer le pod
* Permet à Spring Boot de finir les requêtes

### 🔹 `preStop`

* Donne un petit buffer (ex: 10s)
* Permet au load balancer d’arrêter d’envoyer du trafic

### 🔹 `readinessProbe`

* "Est-ce que je peux recevoir du trafic ?"
* Si FAIL → retiré du load balancer

### 🔹 `livenessProbe`

* "Est-ce que je suis vivant ?"
* Si FAIL → Kubernetes redémarre le pod

---

## ⚠️ Erreurs fréquentes

* ❌ oublier `/actuator/health/readiness`
* ❌ délais trop courts → redémarrages en boucle
* ❌ pas de `terminationGracePeriodSeconds` → requêtes coupées brutalement

# ⚙️ Pipeline CI/CD (exemple GitHub Actions)

Voici un pipeline complet :

name: Build & Deploy to EKS

on:
push:
branches: [ "main" ]

jobs:
build-deploy:
runs-on: ubuntu-latest

```
steps:
  - name: Checkout
    uses: actions/checkout@v4

  - name: Setup Java
    uses: actions/setup-java@v4
    with:
      distribution: 'temurin'
      java-version: '21'

  - name: Build JAR
    run: mvn clean package -DskipTests

  - name: Login to ECR
    uses: aws-actions/amazon-ecr-login@v2

  - name: Build & Push Docker image
    run: |
      IMAGE=ton-compte.dkr.ecr.ca-central-1.amazonaws.com/springboot-app:${{ github.sha }}
      docker build -t $IMAGE .
      docker push $IMAGE
      echo "IMAGE=$IMAGE" >> $GITHUB_ENV

  - name: Update kubeconfig
    run: aws eks update-kubeconfig --region ca-central-1 --name ton-cluster

  - name: Deploy to Kubernetes
    run: |
      sed -i "s|IMAGE_PLACEHOLDER|$IMAGE|g" k8s/deployment.yaml
      kubectl apply -f k8s/
```

# 🧭 À quoi sert un Service ?

Un `Deployment` crée des pods **éphémères** (IP qui changent).

Le `Service` fournit :

* une **IP stable**
* un **load balancing interne**
* un point d’accès pour d’autres services ou un Ingress

👉 Sans Service :

* ton app n’est pas accessible (ou fragile via IP directe)

---

# 🧩 Exemple concret de `service.yaml`

Voici le minimum propre pour Spring Boot :

```yaml
apiVersion: v1
kind: Service
metadata:
  name: springboot-app
spec:
  selector:
    app: springboot-app
  ports:
    - protocol: TCP
      port: 80
      targetPort: 8080
  type: ClusterIP
```

---

# 🔍 Explication rapide

### 🔹 `selector`

Doit matcher ton Deployment :

```yaml
labels:
  app: springboot-app
```

---

### 🔹 `port` vs `targetPort`

* `port: 80` → port exposé dans Kubernetes
* `targetPort: 8080` → port réel de ton app Spring Boot

---

### 🔹 `type: ClusterIP`

* accès **interne seulement**
* parfait pour architecture microservices ou via Ingress
