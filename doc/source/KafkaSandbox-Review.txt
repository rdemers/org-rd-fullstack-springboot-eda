## Le vrai problème: la classe est une « God class »

`KafkaSandbox` mélange **au moins 7 responsabilités**:

1. Lifecycle du broker embedded
2. Cache producers / templates
3. Setup des listeners + error handlers + DLT
4. Administration (AdminClient, topics)
5. Offsets management (`seekConsumerGroupOffsets`)
6. Métriques / dashboard (`calculateConsumerGroupsLag`, `getDashboardData`)
7. Registre de monitors

500+ lignes, 6 caches/registres, synchronisation globale sur `this`. Chaque appel `synchronized` bloque les *autres* opérations non reliées. Par exemple, `getProducer` et `setupMessageListener` se bloquent mutuellement alors qu'ils touchent des caches différents.

**Suggestion**: extraire au moins:
- `KafkaLifecycleManager` (broker + AdminClient)
- `ProducerRegistry` (cache producers/templates)
- `ListenerRegistry` (containers + error handlers)
- `KafkaDashboardService` (dashboard + lag calculations)

Le sandbox devient alors une **façade** légère qui compose ces services.

## Points techniques à revoir

### 1. `seekConsumerGroupOffsets` — le `poll()` est un hack

```java
consumer.seek(tp, Math.min(offset, end));
// ...
consumer.poll(Duration.ofMillis(...));  // ← sert juste à "amorcer" la position
consumer.commitSync();
```

Le `poll()` peut **consommer réellement un record** selon le timing, ce qui n'est probablement pas voulu. La forme canonique est:

```java
Map<TopicPartition, OffsetAndMetadata> toCommit = new HashMap<>();
for (TopicPartition tp : tps) {
    long requested = partitionOffsets.get(tp.partition());
    long end = consumer.endOffsets(List.of(tp)).get(tp);
    toCommit.put(tp, new OffsetAndMetadata(Math.min(requested, end)));
}
consumer.commitSync(toCommit);
```

Pas besoin de `seek`, pas besoin de `poll`. Plus simple, plus sûr.

### 2. Les clés de cache avec postfix String

```java
new ProducerCacheKey(topicName, indTrxFlag ? "_trx" : "_no_trx")
```

Le record accepterait directement un `boolean`:

```java
private record ProducerCacheKey(String topicName, boolean transactional) {}
```

Pas besoin de constantes `CST_TRX_TRX_POSTFIX` / `CST_TRX_NO_TRX_POSTFIX`.

### 3. `TRANSACTIONAL_ACK_MODES` — couplage discutable

```java
KafkaTemplate<K, V> dltTemplate = getKafkaTemplate(
    dltTarget, TRANSACTIONAL_ACK_MODES.contains(ackMode));
```

Ici tu décides si le template DLT est transactionnel en fonction de l'*ackMode du consumer*. Ce lien n'est pas évident. Dans un vrai pipeline exactly-once, ce qu'il faut c'est un `KafkaTransactionManager` qui partage la transaction entre consumer offsets et producer DLT — juste rendre le template transactionnel ne suffit pas à garantir l'atomicité. Soit tu vas jusqu'au bout (tx manager), soit tu laisses non-transactionnel, mais le compromis actuel est bancal.

### 4. `stop()` avec `Thread.sleep` — fragile

```java
listenerCache.values().forEach(lc -> {
    if (!lc.isRunning()) return;
    Thread.sleep(KafkaConstant.CST_POLL_DURATION);
});
```

Attendre un délai fixe pour "drainer" les listeners est brittle. `ConcurrentMessageListenerContainer` a `stop(Runnable callback)` qui te notifie quand c'est vraiment stoppé. Sinon `stop()` est déjà synchrone dans la plupart des cas.

### 5. 4 overloads de `setupMessageListener`

Les 4 premières versions sont du sucre syntaxique. Un **builder** interne simplifierait:

```java
sandbox.listener(topicName)
       .groupId("my-group")
       .concurrency(3)
       .ackMode(AckMode.MANUAL_IMMEDIATE)
       .start(listener);
```

Une seule méthode publique, et les valeurs par défaut sont explicites à la lecture.

### 6. `getDefaultErrorHandler()` avec effet de bord

Un *getter* qui **crée un topic** si nécessaire, c'est surprenant. Renomme-le `getOrCreateDefaultErrorHandler()` ou bouge la création ailleurs (à `start()` par exemple, si `autoCreateTopic` est vrai).

### 7. `getDashboardData() throws Exception`

Trop générique. Les `.get()` sur futures peuvent throw `InterruptedException` et `ExecutionException` — déclare-les spécifiquement, ou wrappe dans une exception métier.

### 8. `Builder.clusters()` deprecated + no-op

Si c'est cassé, **retire la méthode** plutôt que de la marquer `@Deprecated` avec un warning. Un appelant qui passe `3` aura une fausse impression. Au minimum, throw une `UnsupportedOperationException` si la valeur ≠ 1.

### 9. Propriétés broker hardcodées

```java
"log.retention.hours", "1",
```

Pour un sandbox c'est OK, mais si quelqu'un veut tester la rétention, il est coincé. Rends-les override-ables via le builder.

Excellente question — c'est un cas classique en prod, et oui, c'est tout à fait possible **sans déclencher de rebalance**. La clé c'est de comprendre la différence entre:

- `container.stop()` → le consumer **quitte le groupe** → **rebalance** 🚫
- `container.pause()` → le consumer **reste dans le groupe**, continue à faire des heartbeats, mais arrête de consommer → **pas de rebalance** ✅

## Comment ça marche sous le capot

Spring Kafka expose `pause()` / `resume()` sur le `MessageListenerContainer`. En interne:

1. Le container marque un flag "paused".
2. Au prochain `poll()`, il appelle `consumer.pause(assignedPartitions)` sur les partitions assignées.
3. **Le poll continue à tourner** — c'est crucial parce que c'est ce qui envoie les heartbeats au group coordinator. Le consumer reste donc « vivant » du point de vue du groupe.
4. Simplement, `poll()` retourne zéro records tant que les partitions sont pause.

Quand tu fais `resume()`, le flag tombe, `consumer.resume(...)` est appelé, et les messages recommencent à arriver.

## Le cas d'usage exact que tu décris

Multiprocessus, incident détecté, tu veux suspendre la conso **sur cette instance** sans que les autres récupèrent ses partitions (ce qui causerait une tempête de rebalances):

```java
@Component
public class MonListener {

    private static final String LISTENER_ID = "monListener";

    private final KafkaListenerEndpointRegistry registry;

    public MonListener(KafkaListenerEndpointRegistry registry) {
        this.registry = registry;
    }

    @KafkaListener(id = LISTENER_ID, topics = "mon-topic")
    public void onMessage(ConsumerRecord<String, String> record) {
        // traitement
    }

    public void pauseOnIncident() {
        MessageListenerContainer c = registry.getListenerContainer(LISTENER_ID);
        if (c != null && c.isRunning() && !c.isPauseRequested()) {
            c.pause();   // ← pas de rebalance
        }
    }

    public void resumeAfterIncident() {
        MessageListenerContainer c = registry.getListenerContainer(LISTENER_ID);
        if (c != null && c.isPauseRequested()) {
            c.resume();
        }
    }
}
```

Deux détails importants:

1. **Donne un `id` explicite à ton `@KafkaListener`** — sinon tu récupères l'auto-généré (`org.springframework.kafka.KafkaListenerEndpointContainer#0`), fragile si tu réordonnes tes listeners.

2. **`isPauseRequested()` vs `isContainerPaused()`** — le premier te dit « j'ai demandé pause », le second te dit « le consumer a effectivement appliqué la pause au prochain poll ». Il y a un petit délai entre les deux. Pour juste décider s'il faut pause/resume, `isPauseRequested()` suffit.

## Attention à `max.poll.interval.ms`

C'est le **seul vrai piège**. Même en pause, le consumer continue à poller — mais si entre deux polls tu dépasses `max.poll.interval.ms` (5 min par défaut), le broker considère le consumer mort et **déclenche un rebalance**. En pause normale ce n'est jamais un souci (le container poll en continu), mais si ton incident bloque aussi la thread du listener (par exemple un lock deadlock), tu peux finir par rater le deadline.

Si tu as peur de ce cas, augmente `max.poll.interval.ms` pour tes listeners critiques, ou mets en place un healthcheck qui détecte ce scénario.

## Pour aller plus loin: pause au niveau partition

Si tu veux pauser **une seule partition** (par exemple parce qu'un traitement sur une clé spécifique échoue), il y a `container.pausePartition(TopicPartition)`. Même principe — aucun rebalance, juste cette partition qui arrête de livrer.

## Bonus: pause automatique sur erreur

Si ton besoin c'est « en cas d'erreur répétée, mets en pause tout seul », Spring a `DefaultErrorHandler` avec `addRetryableExceptions(...)` et surtout **`ContainerPausingBackOffHandler`** (depuis Spring Kafka 3.0). Tu peux le brancher à ton error handler pour que le container se pause automatiquement entre deux tentatives, au lieu de `Thread.sleep` sur la thread du listener (ce qui risque justement de déclencher `max.poll.interval.ms`).

```java
DefaultErrorHandler handler = new DefaultErrorHandler(recoverer, backOff);
handler.setBackOffHandler(new ContainerPausingBackOffHandler(/* ... */));
```

Très utile si ton « incident » c'est une dépendance externe (DB, API) qui flappe — le container se pause, laisse passer le backoff, puis reprend, **sans jamais quitter le groupe**.

## La réponse courte

**Non, par défaut ton `@KafkaListener` va redémarrer en état actif (non-pausé)** quand le nouveau pod se lance. L'état « paused » **ne persiste pas** à travers un redémarrage du pod.

Voici pourquoi — et pourquoi c'est à la fois une bonne et une mauvaise nouvelle.

## Ce qui se passe vraiment

L'état de pause est **en mémoire** dans le `MessageListenerContainer` Spring. Quand ton pod est détruit:

1. Le pod meurt → le container meurt avec lui → l'état « paused » disparaît.
2. Kubernetes démarre un nouveau pod ailleurs.
3. Spring Boot démarre, le `KafkaListenerEndpointRegistry` construit un **nouveau** container.
4. Ce nouveau container démarre par défaut en état **running** (pas pausé).
5. Le consumer rejoint le groupe → **rebalance** (inévitable, tu le sais déjà).
6. Il commence à consommer immédiatement.

**Donc le scénario que tu redoutes en surface — « ça reste pausé au redémarrage » — n'arrive pas par défaut.** Mais l'inverse est souvent pire:

## Le vrai problème: tu voulais peut-être qu'il reste pausé

Si tu as pausé le listener parce que la dépendance downstream est down (API tierce KO, DB en maintenance, circuit breaker ouvert), et que ton pod redémarre pendant l'incident... **le nouveau pod va se mettre à consommer tout de suite** et **re-échouer**, potentiellement en poussant vers la DLT ou en saturant la downstream.

C'est souvent ça le vrai risque avec les pods éphémères + pause.

## Les solutions

### Option 1 — `autoStartup = false` + healthcheck applicatif

Empêche le listener de démarrer automatiquement, et laisse ton code décider:

```java
@KafkaListener(id = "monListener", topics = "mon-topic", autoStartup = "false")
public void onMessage(ConsumerRecord<String, String> record) { ... }
```

Puis dans un `ApplicationRunner` ou un readiness probe custom:

```java
@Component
public class ListenerBootstrap implements ApplicationRunner {

    private final KafkaListenerEndpointRegistry registry;
    private final DownstreamHealthCheck health;

    @Override
    public void run(ApplicationArguments args) {
        if (health.isHealthy()) {
            registry.getListenerContainer("monListener").start();
        } else {
            log.warn("Downstream KO — listener stays stopped until health recovers.");
            // Schedule a retry loop ou laisse un autre mécanisme le démarrer.
        }
    }
}
```

**Limite**: ici j'utilise `start()` / `stop()`, pas `pause()` / `resume()`. `stop()` va quitter le groupe et causer un rebalance au démarrage — mais ce rebalance va de toute façon arriver puisque le pod a redémarré. Donc on ne perd rien.

### Option 2 — Persister l'état de pause

Stocker l'intention « listener doit être pausé » dans un endroit durable (Redis, ConfigMap, base de données, un record Kafka sur un topic de contrôle). Au démarrage, le nouveau pod lit cet état et décide:

```java
@EventListener(ApplicationReadyEvent.class)
public void onStartup() {
    if (pauseStateRepository.isListenerPaused("monListener")) {
        MessageListenerContainer c = registry.getListenerContainer("monListener");
        // Démarre puis pause immédiatement.
        c.start();
        c.pause();
    }
}
```

Ça marche mais c'est un peu gauche — tu fais `start()` pour rejoindre le groupe, puis `pause()` tout de suite. Le consumer est dans le groupe (heartbeats OK, partitions assignées) mais ne consomme rien. C'est exactement ce que tu veux fonctionnellement, mais conceptuellement c'est deux états.

### Option 3 — Ne pas démarrer tant que la cause est là

Si ton incident est détectable (health check vers la downstream), `autoStartup=false` + une boucle qui démarre le container quand le check passe est souvent **plus propre** qu'un mécanisme de pause persistante. Tu traites l'incident comme « cette instance n'est pas prête à consommer », pas comme « cette instance est en pause ».

### Option 4 — Le pattern Kubernetes-natif: readiness probe

Expose ton état via un endpoint Spring Boot Actuator custom, et configure ta **readiness probe** pour qu'elle ne passe pas tant que la downstream est KO. Kubernetes ne routera pas de trafic entrant, mais plus important: **si tu couples `autoStartup=false` avec un démarrage conditionnel basé sur la readiness**, le listener ne démarre jamais tant que le pod n'est pas « ready ».

```java
@Component
public class KafkaListenerController implements ApplicationListener<AvailabilityChangeEvent<ReadinessState>> {

    private final KafkaListenerEndpointRegistry registry;

    @Override
    public void onApplicationEvent(AvailabilityChangeEvent<ReadinessState> event) {
        MessageListenerContainer c = registry.getListenerContainer("monListener");
        if (c == null) return;

        if (event.getState() == ReadinessState.ACCEPTING_TRAFFIC) {
            if (!c.isRunning()) c.start();
        } else {
            if (c.isRunning()) c.stop();  // ou c.pause() — voir ci-dessous
        }
    }
}
```

## `stop()` vs `pause()` au redémarrage — le vrai trade-off

Dans le contexte « mon pod redémarre et je veux qu'il soit « retenu » »:

| Approche | Comportement | Rebalance |
|---|---|---|
| `autoStartup=false` + `start()` conditionnel | Ne rejoint le groupe que quand prêt | Oui, un seul, au `start()` |
| Start immédiat puis `pause()` persistée | Rejoint le groupe puis ne consomme pas | Oui, au démarrage, comme d'habitude |

Le premier est **plus propre**: tant que le listener n'est pas démarré, **il n'est pas dans le groupe du tout**. Pas d'heartbeats, pas de partitions assignées, pas de faux positifs dans tes dashboards de lag. Les partitions vont à d'autres pods sains qui peuvent vraiment consommer.

Le second introduit un état bizarre où le pod a des partitions assignées mais ne les traite pas — potentiellement du lag qui s'accumule **sur ces partitions précisément**, pendant que d'autres pods sains auraient pu les prendre.

## Ma recommandation pour ton contexte EKS

Pour pods éphémères avec incidents sur dépendances downstream:

1. **`autoStartup = false`** sur les `@KafkaListener` concernés.
2. Un composant qui écoute `ReadinessState` ou fait un healthcheck périodique.
3. **`start()` / `stop()` conditionnel** plutôt que `pause()` / `resume()`.
4. Accepte le rebalance au démarrage (il est inévitable avec des pods éphémères de toute façon).
5. Garde `pause()` pour les incidents **courts** (< quelques minutes) où tu veux éviter la DLT en rafale mais pas perdre les partitions.

La règle empirique que j'utilise: **`pause()` pour des incidents transitoires, `stop()` + restart conditionnel pour des incidents qui pourraient survivre à un redémarrage de pod**. En EKS avec des pods qui bougent, la deuxième catégorie est la bonne valeur par défaut.

## 1. La dépendance à surveiller

```java
@Component
public class DownstreamHealth {

    private static final Logger logger = LoggerFactory.getLogger(DownstreamHealth.class);

    private final DataSource dataSource;
    private volatile boolean healthy = false;

    public DownstreamHealth(DataSource dataSource) {
        this.dataSource = dataSource;
    }

    /**
     * Ping léger — à appeler périodiquement par le probe et par le contrôleur.
     */
    public boolean checkNow() {
        try (Connection c = dataSource.getConnection();
             Statement s = c.createStatement()) {
            s.setQueryTimeout(2);
            s.execute("SELECT 1");
            if (!healthy) {
                logger.info("Downstream recovered.");
            }
            healthy = true;
        } catch (Exception ex) {
            if (healthy) {
                logger.warn("Downstream became unhealthy: {}.", ex.toString());
            }
            healthy = false;
        }
        return healthy;
    }

    public boolean isHealthy() {
        return healthy;
    }
}
```

## 4. Le contrôleur de lifecycle

C'est ici que toute la logique vit. Il écoute les changements d'état de readiness **et** fait une vérification périodique au cas où.

```java
@Component
public class KafkaListenerLifecycleController {

    private static final Logger logger = LoggerFactory.getLogger(KafkaListenerLifecycleController.class);

    private final KafkaListenerEndpointRegistry registry;
    private final DownstreamHealth health;
    private final ApplicationAvailability availability;

    public KafkaListenerLifecycleController(KafkaListenerEndpointRegistry registry,
                                            DownstreamHealth health,
                                            ApplicationAvailability availability) {
        this.registry = registry;
        this.health = health;
        this.availability = availability;
    }

    /**
     * Au démarrage, on évalue une fois et on démarre si tout va bien.
     * Ne démarre PAS en mode "start then pause" — on attend d'être sûr.
     */
    @EventListener(ApplicationReadyEvent.class)
    public void onStartup() {
        if (health.checkNow()) {
            startListener();
        } else {
            logger.warn("Application ready but downstream KO. Listener stays stopped.");
            // Le reconcile() périodique le démarrera quand la downstream reviendra.
        }
    }

    /**
     * Réagit aux changements de readiness émis par Spring Boot (ex: le HealthIndicator
     * passe DOWN, ça fait basculer la readiness, on reçoit l'évènement ici).
     */
    @EventListener
    public void onReadinessChange(AvailabilityChangeEvent<ReadinessState> event) {
        logger.info("Readiness changed to {}.", event.getState());
        reconcile();
    }

    /**
     * Filet de sécurité — on re-check toutes les 10 secondes au cas où on aurait raté
     * un évènement, ou pour relancer le listener quand la downstream revient.
     */
    @Scheduled(fixedDelay = 10_000L, initialDelay = 10_000L)
    public void periodicReconcile() {
        health.checkNow();  // refresh du flag
        reconcile();
    }

    // ---------------------------------------------------------------
    // Logique de réconciliation
    // ---------------------------------------------------------------

    private void reconcile() {
        boolean shouldRun = health.isHealthy()
            && availability.getReadinessState() == ReadinessState.ACCEPTING_TRAFFIC;

        MessageListenerContainer container = registry.getListenerContainer(OrderListener.LISTENER_ID);
        if (container == null) {
            logger.warn("Listener container '{}' not found.", OrderListener.LISTENER_ID);
            return;
        }

        if (shouldRun && !container.isRunning()) {
            startListener();
        } else if (!shouldRun && container.isRunning()) {
            stopListener();
        }
    }

    private void startListener() {
        MessageListenerContainer container = registry.getListenerContainer(OrderListener.LISTENER_ID);
        if (container != null && !container.isRunning()) {
            logger.info("Starting Kafka listener '{}'.", OrderListener.LISTENER_ID);
            container.start();
        }
    }

    private void stopListener() {
        MessageListenerContainer container = registry.getListenerContainer(OrderListener.LISTENER_ID);
        if (container != null && container.isRunning()) {
            logger.info("Stopping Kafka listener '{}' — downstream unhealthy.", OrderListener.LISTENER_ID);
            container.stop();   // quitte le groupe — rebalance, mais c'est voulu
        }
    }
}
```

N'oublie pas `@EnableScheduling` sur une config class pour que `@Scheduled` fonctionne:

```java
@Configuration
@EnableScheduling
public class SchedulingConfig { }
```

## 5. Shutdown propre

Quand Kubernetes envoie `SIGTERM` au pod (déplacement, scale-in, rolling update), il faut que ton listener arrête **proprement** avant que le JVM meure, sinon tu risques de redelivery inutile:

```yaml
spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s
  kafka:
    listener:
      # Laisse au container le temps de finir son poll courant avant d'être tué.
      shutdown-timeout: 10000
server:
  shutdown: graceful
```

Et dans ton Deployment Kubernetes:

```yaml
spec:
  terminationGracePeriodSeconds: 45   # > timeout-per-shutdown-phase
```

Spring Boot prend soin du `container.stop()` à l'arrêt, tu n'as rien à coder pour ça.

## Ce que ça donne en pratique

**Scénario 1: pod démarre avec DB OK**
1. Spring démarre, listener est `autoStartup=false` → pas dans le groupe.
2. `ApplicationReadyEvent` fire → `checkNow()` → healthy → `start()`.
3. Listener rejoint le groupe, consomme. Un seul rebalance, normal.

**Scénario 2: pod démarre avec DB KO**
1. Spring démarre, listener pas démarré.
2. `onStartup()` → `checkNow()` → KO → on ne démarre pas.
3. `/actuator/health/readiness` retourne DOWN → Kubernetes ne route pas de trafic.
4. Le `periodicReconcile` tourne toutes les 10s.
5. Quand DB revient → `checkNow()` → OK → `start()` → rejoint le groupe.

**Scénario 3: pod tourne, DB tombe pendant la vie du pod**
1. `periodicReconcile` détecte → `checkNow()` KO.
2. `reconcile()` voit `shouldRun=false` et le container running → `stop()`.
3. Listener quitte le groupe, ses partitions sont prises par les autres pods.
4. Quand la DB revient sur ce pod → `start()`, retour dans le groupe.

**Scénario 4: pod tué pendant incident, redémarre ailleurs**
1. Nouveau pod, DB KO → ne démarre pas le listener.
2. Kubernetes voit `readiness` DOWN, ne route pas de trafic.
3. Pas de consommation erronée, pas de cascade d'échecs.
4. Quand tout va bien → démarre automatiquement.

## Pourquoi `stop()` et pas `pause()`

Tu as raison que `stop()` déclenche un rebalance. Mais dans ce pattern:

- Au démarrage, de toute façon tu rejoins le groupe → rebalance inévitable.
- En run-time, si la DB tombe **sur tous les pods** en même temps, tous vont `stop()` → les partitions n'ont nulle part où aller → c'est OK, elles reviendront à l'`start()`.
- Si la DB tombe **sur ce pod seulement** (cas rare mais possible — network partition, config locale), les partitions vont ailleurs sur des pods sains → **c'est ce qu'on veut**.

`pause()` aurait gardé les partitions sur ce pod malade, et elles auraient accumulé du lag sans être traitées. Mauvaise propriété.

## Une variante qui vaut la peine

Si tu veux un filet en plus, expose un endpoint admin pour forcer start/stop manuellement:

```java
@RestController
@RequestMapping("/admin/listener")
public class ListenerAdminController {

    private final KafkaListenerEndpointRegistry registry;

    public ListenerAdminController(KafkaListenerEndpointRegistry registry) {
        this.registry = registry;
    }

    @PostMapping("/{id}/start")
    public ResponseEntity<String> start(@PathVariable String id) {
        var c = registry.getListenerContainer(id);
        if (c == null) return ResponseEntity.notFound().build();
        c.start();
        return ResponseEntity.ok("started");
    }

    @PostMapping("/{id}/stop")
    public ResponseEntity<String> stop(@PathVariable String id) {
        var c = registry.getListenerContainer(id);
        if (c == null) return ResponseEntity.notFound().build();
        c.stop();
        return ResponseEntity.ok("stopped");
    }

    @GetMapping("/{id}/state")
    public ResponseEntity<Map<String, Object>> state(@PathVariable String id) {
        var c = registry.getListenerContainer(id);
        if (c == null) return ResponseEntity.notFound().build();
        return ResponseEntity.ok(Map.of(
            "running", c.isRunning(),
            "paused", c.isPauseRequested(),
            "containerPaused", c.isContainerPaused()
        ));
    }
}
```

Très utile pendant un incident pour forcer la main sans redéployer. Mets-le derrière du basic auth / de l'auth interne évidemment.

**Point de vigilance que j'ai pris soin de câbler correctement:**

La liveness probe n'est **pas** liée à la health downstream — seulement la readiness. C'est crucial: si tu mets la downstream dans la liveness, Kubernetes va tuer tes pods à chaque hoquet de la DB, et tu vas avoir des rolling restarts inutiles. La readiness est le bon niveau pour « ce pod n'est pas prêt, retire-le du trafic ».

Le contrôleur de lifecycle est **synchronized** sur un lock dédié parce que trois déclencheurs peuvent l'appeler en même temps (ApplicationReadyEvent, AvailabilityChangeEvent, @Scheduled) — sans ça tu pourrais avoir des `start()`/`stop()` qui se marchent dessus.

Si tu as besoin d'adapter pour plusieurs listeners, la logique est facilement extensible: le `reconcile()` tourne sur un seul `LISTENER_ID` pour l'exemple, mais tu peux le généraliser en itérant sur `registry.getAllListenerContainers()` avec une règle par listener.
