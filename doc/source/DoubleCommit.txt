@Configuration
public class KafkaConfig {

    @Bean
    public KafkaTransactionManager<String, String> kafkaTransactionManager(
            ProducerFactory<String, String> pf) {
        return new KafkaTransactionManager<>(pf);
    }

    @Bean
    public PlatformTransactionManager chainedTransactionManager(
            KafkaTransactionManager<String, String> kafkaTM,
            JpaTransactionManager jpaTM) {
        return new ChainedKafkaTransactionManager<>(kafkaTM, jpaTM);
    }

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String>
    kafkaListenerContainerFactory(
            ConsumerFactory<String, String> consumerFactory,
            PlatformTransactionManager chainedTransactionManager) {

        ConcurrentKafkaListenerContainerFactory<String, String> factory =
                new ConcurrentKafkaListenerContainerFactory<>();

        factory.setConsumerFactory(consumerFactory);

        factory.getContainerProperties().setAckMode(ContainerProperties.AckMode.MANUAL);
        factory.getContainerProperties().setTransactionManager(chainedTransactionManager);

        return factory;
    }
}

@Component
public class OrderConsumer {

    @Autowired
    private OrderService orderService;

    @KafkaListener(
        topics = "order-topic",
        containerFactory = "kafkaListenerContainerFactory"
    )
    @Transactional  // << IMPORTANT: uses ChainedKafkaTransactionManager
    public void listen(String message, Acknowledgment ack) {

        // --- 1) Save into Database ---
        orderService.saveOrder(message);

        // --- 2) Commit Kafka offset only after DB commit is safe ---
        ack.acknowledge();

        // If anything throws → both DB and Kafka offset rollback
    }
}

In this configuration, the producer doesn’t wait for a reply from the broker. It assumes that the message is sent successfully. If something goes wrong and the broker doesn’t receive the message, the producer is unaware of it, and the message is lost.

However, because the producer doesn’t wait for any response from the server, it can send messages as fast as the network supports, achieving high throughput.

---------------
We can instead set property acks to 1:

The producer receives a successful response from the broker the moment the leader replica receives the message. If the message can’t be written to the leader due to any error, the producer receives an error response and retries sending the message, avoiding the potential loss of data.

We can still lose a message for other reasons, such as if the leader crashes before the system replicates the latest messages to the new leader.

-------------------------
We can also set property acks to all:

The producer receives a success response from the broker once all in-sync replicas receive the message. This is the safest mode since we can make sure more than one broker has the message and that the message survives even in case of a crash.

The main difference is that MANUAL commits are processed in a batch after the listener has processed all records in a poll, while MANUAL_IMMEDIATE commits the offset immediately when the listener calls acknowledge() on the consumer thread. This means MANUAL_IMMEDIATE provides more immediate confirmation of a successful commit, whereas MANUAL defers the commit to the end of the batch to optimize performance. 
MANUAL
Commit timing: The acknowledgment is queued by the listener. Offsets are committed only after the listener has finished processing all records from a poll() operation.
Benefit: Groups acknowledgments to reduce the overhead of frequent commits, potentially improving throughput.
Scenario: Suitable when immediate acknowledgment isn't critical and you want to batch commits for efficiency. 
MANUAL_IMMEDIATE
Commit timing: The offset is committed immediately when the acknowledge() method is called on the listener thread.
Benefit: Provides more immediate feedback on the commit's success. If using synchronous commits, it allows the application to catch exceptions from the commit itself, enabling immediate error handling.
Scenario: Ideal for situations where you need to know immediately if a message has been successfully committed, even if it means slightly more overhead.
