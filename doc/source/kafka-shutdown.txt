///////////////////////////////////////////////////////////////////////////////
`consumer.wakeup()` est **la méthode officielle fournie par Kafka** pour interrompre proprement un thread en train de faire `poll()`.

C’est *LA* façon recommandée pour sortir proprement d’un consumer Kafka bloqué sur un poll long.

---

# 🧠 **Que fait exactement `wakeup()` ?**

### ✔ 1. **Interrompt immédiatement `poll()`**

Si ton consumer est bloqué ici :

```java
consumer.poll(Duration.ofSeconds(10));
```

`wakeup()` va provoquer une exception **`WakeupException`** dans ce thread.
Cela permet de reprendre la main **sans attendre la fin du poll**.

---

### ✔ 2. **Ne ferme PAS le consumer**

Ça ne shutdown pas Kafka, ça ne ferme pas la connexion.
Ça sert juste à sortir du polling.

➡️ Ensuite c’est à toi de faire le cleanup :

```java
consumer.close();
```

---

### ✔ 3. **Thread-safe**

Tu peux appeler `wakeup()` depuis **un autre thread**, par exemple :

* un listener Spring `ContextClosedEvent`
* un `@PreDestroy`
* un shutdown hook
* un thread d'administration

C’est fait pour ça.

---

# 🧪 Exemple ultra-classique

```java
public class MyKafkaConsumer implements Runnable {

    private final KafkaConsumer<String, String> consumer;
    private volatile boolean running = true;

    public MyKafkaConsumer(KafkaConsumer<String, String> consumer) {
        this.consumer = consumer;
    }

    @Override
    public void run() {
        try {
            while (running) {
                ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(1000));
                // ... traitement
            }
        } catch (WakeupException e) {
            if (running) throw e; // erreur si ce n'est pas un shutdown volontaire
        } finally {
            consumer.close(); // fermeture propre, commit final
        }
    }

    public void shutdown() {
        running = false;
        consumer.wakeup();   // interrompt poll()
    }
}
```

➡️ `shutdown()` peut être appelé depuis un événement Spring :
`ContextClosedEvent → consumer.wakeup()`.

---

# 🟦 **Pourquoi ne pas utiliser `interrupt()` ?**

Parce que `poll()` **ignore** les interruptions de thread.
Kafka a besoin d'un mécanisme spécial → `wakeup()`.

---

# ✅ En résumé

`wakeup()` permet de :

* interrompre proprement un `poll()` bloquant
* sortir la boucle de consommation
* effectuer un `close()` clean
* ne pas bloquer le graceful shutdown

C’est le mécanisme prévu par Kafka pour gérer les arrêts contrôlés.

Bonne question — et **oui**, si tu fais juste `return` dans un `@KafkaListener` **sans ack explicite**, le message sera considéré comme *consommé* selon ta configuration Kafka.

➡️ **Le risque :**

* si `enable.auto.commit=true` → auto commit → **message perdu**
* si `AckMode` = `BATCH`, `TIME`, `COUNT` → le commit peut arriver quand même → **message perdu**

Donc : ❌ ne jamais “juste return” lors d’un shutdown si tu veux rejouer le message.

---

# ✔️ SOLUTION : empêcher la consommation **ET NE PAS ACK**

Pour ça, il faut :

1. Utiliser un `AckMode` qui nécessite un **ack explicite**
2. **Ne pas ack** le message si un shutdown est en cours
3. Laisser Kafka le redélivrer au prochain démarrage

---

# 🧩 1. Désactiver `autoCommit` + activer ACK MANUEL

Configuration :

```java
spring.kafka.consumer.enable-auto-commit=false
spring.kafka.listener.ack-mode=MANUAL
```

ou en Java :

```java
factory.getContainerProperties().setAckMode(ContainerProperties.AckMode.MANUAL);
```

---

# 🧩 2. Dans ton listener, si shutdown → **ne PAS ack**, juste sortir

Exemple complet :

```java
@Component
@RequiredArgsConstructor
public class MyKafkaListener {

    private final ShutdownFlag shutdownFlag;

    @KafkaListener(topics = "demo", containerFactory = "kafkaListenerContainerFactory")
    public void listen(String msg, Acknowledgment ack) {

        if (shutdownFlag.isShuttingDown()) {
            // Pas d’ACK → Kafka redélivrera le message au prochain démarrage
            log.info("Shutdown détecté → je skip MAIS je ne commit PAS");
            return;
        }

        // --- logique métier ---
        traiterMessage(msg);

        // ack explicite
        ack.acknowledge();
    }
}
```

### 👉 Résultat :

* si shutdown : message **non acké**, donc **non committé**
* Kafka redélivre à la prochaine exécution

➡️ **Tu ne perds aucun message.**

# 🟥 **5. “Drain first, stop later”**

Méthode avancée utilisée dans les gros environnements Kafka (banques, telco, cloud) :

Processus :

1. Tu empêche le consumer de prendre de **nouveaux** messages → `pause()`
2. Tu termines les messages en cours
3. Tu arrêtes le container
4. Tu fais un commit final (ou non, selon la logique)

Exemple :

```java
container.pause();

try {
    // attendre la fin de la boucle de poll
    Thread.sleep(1000);
} catch (InterruptedException ignored) {}

container.stop();
```

➡️ Permet d’éviter d'abandonner un message **durant** un traitement métier.

# 🟫 **6. “Commit On Success, SeekOnFailure” (Error Handler)**

```java
        // on failure → repositionnement
```

➡️ En cas de shutdown brutal pendant le traitement → pas de commit.

➡️ À combiner avec shutdownFlag.

* arrêter graduellement Kafka
* terminer les messages en cours avant le `kill -9`

# 🟫 **8. Utiliser un “BatchListener”**

Si tes messages sont consommés par lot :

* traitement batch
* commit groupé après la logique métier
* si shutdown → tout le lot est **non acké** → reprocess complet

```java
@KafkaListener(batch = "true")

    if (shutdownFlag.isShuttingDown()) {
        return; // non ack → tout le batch est rejoué
    }

    // traitement du lot…
    ack.acknowledge();
}
```

➡️ Très efficace pour throughput élevé + fiabilité forte.

* jamais perdu
* reprocessé plus tard (ou analysé)

```java
DeadLetterPublishingRecoverer recoverer = new DeadLetterPublishingRecoverer(template);
```

# 🟦 **10. Utiliser des Transactions Kafka (Exactly-Once)**

Le nirvana pour les systèmes très sensibles :

* listener transactionnel
* processing transactionnel
* commit des offsets **dans la même transaction**

Si shutdown avant la fin → rollback complet → message rejoué.

Setup :


➡️ Ultra-sûr → zéro perte, zéro doublon (exactly-once).

# 🟦 1. OBLIGATOIRE en Kubernetes : **Graceful shutdown K8s bien configuré**

## 1.1 readinessProbe + unpreprared grace period

Avant qu’un pod soit tué, Kubernetes :

1. envoie un **SIGTERM**
2. marque le pod comme **not-ready** (via readinessProbe)
3. arrête d’envoyer du trafic
4. *attend un délai* (`terminationGracePeriodSeconds`)
5. si le pod n’est pas arrêté → `SIGKILL`

Pour Kafka, tu veux **terminer proprement le poll en cours** et **ne plus en prendre de nouveaux**.

👉 Mets **minimum 30 secondes** (ou 60 si ton traitement métier est long) :

```yaml
spec:
  terminationGracePeriodSeconds: 60
```

👉 Ajoute un *preStop hook* (ceci est crucial) :

```yaml
lifecycle:
  preStop:
    exec:
      command: ["sh", "-c", "sleep 10"]
```

➡️ Donne le *temps* à Spring Kafka d’arrêter proprement le consumer.

* **auto-commit désactivé** → zéro perte.
* **max.poll.interval.ms** large pour éviter les rebalance intempestifs si ton traitement métier est long.

# 🟥 4. L'élément clé EKS : **Consumer Group Rebalancing**

Quand un pod meurt :

* Kafka detecte que le consumer n’envoie plus de heartbeat
* les autres pods prennent sa part des partitions (rebalance)
* tout message **non acké** est automatiquement re-delivré

Donc :
👉 **ne jamais ack sur shutdown** = garantie de redelivery.
👉 dans ton cluster 2–6 pods, c’est parfait.

### ✔ Ne JAMAIS utiliser `auto-commit`

### ✔ Pour les traitements longs : `pause()` sur SIGTERM

* pause le container
* terminer le message en cours
* puis shutdown

Si tu veux je peux te donner un code “pause on shutdown”.

# 🟦 1. **Configuration Spring Kafka optimisée pour EKS**

Voici un **application.yml minimal mais ultra-stable**.

## ✔ 1.1 Désactiver auto commit (obligatoire)

```yaml
spring:
  kafka:
    consumer:
      enable-auto-commit: false
```

➡️ évite toute perte en cas de shutdown.

---

## ✔ 1.2 ACK manuel et immédiat

```yaml
spring:
  kafka:
    listener:
      ack-mode: MANUAL_IMMEDIATE
```

* MANUAL_IMMEDIATE = commit dès `ack.acknowledge()`
* meilleure cohérence avec rebalances fréquents en EKS

---

## ✔ 1.3 Mettre un timeout de poll faible : réactivité au SIGTERM

```yaml
spring:
  kafka:
    consumer:
      properties:
        max.poll.records: 50
        max.poll.interval.ms: 300000 # 5 minutes
      fetch-max-wait: 200
```

➡️ `max.poll.records: 50` = Reactivité très rapide au shutdown
➡️ `max.poll.interval.ms: 5 min` = autorise traitements métier raisonnables
➡️ Évite les rebalances intempestifs en cas de traitement un peu plus long

---

## ✔ 1.4 Augmenter la fréquence des heartbeats (super efficace en EKS)

```yaml
spring:
  kafka:
    consumer:
      properties:
        heartbeat.interval.ms: 1000
        session.timeout.ms: 10000
```

Explications :

* heartbeats fréquents → Kafka sait rapidement que le pod vit
* session timeout à 10 sec → quand un pod meurt, rebalance rapide
* pas trop bas pour éviter des rebalances intempestifs

---

## ✔ 1.5 Stop propre du container côté Spring Kafka

```yaml
spring:
  kafka:
    listener:
      no-poll-throttle: 100
```

➡️ Le listener est réveillé *immédiatement* lors d’un SIGTERM → le `poll()` se termine vite.

# 🟧 2. **Code listener stable**

### ✔ Flag global pour shutdown detection

```java
@Component
public class ShutdownFlag {
    private final AtomicBoolean shuttingDown = new AtomicBoolean(false);

    @EventListener
    public void onShutdown(ContextClosedEvent e) {
        shuttingDown.set(true);
    }

    public boolean isShuttingDown() {
        return shuttingDown.get();
    }
}
```

### ✔ Listener Kafka robuste pour EKS

```java
@KafkaListener(
    topics = "my-topic",
    concurrency = "3",          // à ajuster selon pods
    id = "my-consumer"
)
public void consume(String msg, Acknowledgment ack) {

    // 🔥 important : pas d’ACK sur shutdown => pas de perte
    if (shutdownFlag.isShuttingDown()) {
        log.warn("Shutdown détecté → pas d’ACK, redelivery assurée");
        return;
    }

    try {
        // 👉 process métier
        process(msg);
        ack.acknowledge();

    } catch (Exception e) {
        log.error("Erreur métier : message non acké, redelivery", e);
        // ne pas ack => redelivery
    }
}
```

➡️ **NE JAMAIS ack dans un shutdown**.

➡️ Le message revient automatiquement sur un autre pod.

# 🟩 3. **Configuration EKS Kubernetes (indispensable)**

Voici les réglages parfaits pour EKS :

---

## ✔ 3.1 readinessProbe rapide

```yaml
readinessProbe:
  httpGet:
    path: /actuator/health/readiness
    port: 8080
  initialDelaySeconds: 3
  periodSeconds: 2
  failureThreshold: 1
```

➡️ Quand un shutdown commence, readiness tombe à `False` →
Kafka **retire immédiatement** les partitions du pod (rebalance propre).

---

## ✔ 3.2 preStop + grace period suffisante

```yaml
lifecycle:
  preStop:
    exec:
      command: ["sh", "-c", "sleep 10"]
```

```yaml
terminationGracePeriodSeconds: 60
```

➡️ Donne à Spring Kafka **le temps d'arrêter proprement**.

➡️ Evite les `SIGKILL` pendant un poll → source majeure de problèmes Kafka en K8s.

---

## ✔ 3.3 Liveness probe raisonnable

```yaml
livenessProbe:
  httpGet:
    path: /actuator/health/liveness
    port: 8080
  initialDelaySeconds: 15
  periodSeconds: 10
```

➡️ Ne redémarre pas un pod pendant une longue opération métier.

---

## ✔ 3.4 Anti-Affinity (pour éviter que tous les pods meurent d’un coup)

```yaml
affinity:
  podAntiAffinity:
    preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 100
        podAffinityTerm:
          topologyKey: kubernetes.io/hostname
          labelSelector:
            matchLabels:
              app: kafka-consumer
```

➡️ Les pods sont répartis sur différents nodes → moins de rebalances massifs.
